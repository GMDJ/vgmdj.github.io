消息队列学习-RabbitMQ健康检查

发表于 2018-06-18 | 更新于: 2018-06-29 | 分类于 消息队列 | | 阅读次数:
字数统计: 1.4k | 阅读时长 ≈ 0:01

Rest API 监测

需要注意的是RabbitMQ中对外提供的Rest API接口，是Management Plugin所提供的，如果没有安装这个插件的话，则需要使用amqp的方式

  • vhost 检查
    用于检测vhost的运行情况，如果正常，则会收到 “ok”

  • queue检查
    用于获取队列当前信息

AMQP模拟检测

对于AMQP的queue_declare()命令，如果设置passive=True，那么RabbitMQ在收到这个参数时就会明白，此时并不是真正的想要声明一个队列，如果队列已存在的话，会返回队列消息的总数，如果队列不存在的话，则会引发异常。
对于模拟检测来讲，passive一定要设为true，因为这不仅是获取队列消息总数的唯一方法，同时也是因为，如果没有传递该参数的话，检测程序会真实地尝试去声明该队列。

确保消费者运行状态

对于生产者而言，如果发送失败，则说明此时有异常出现，之后就可以尝试进行修复。
但是对于消费者而言，如果中途网络不稳定导致连接断掉，并不能直接就检测出此时的状态。
因此，一般采用的方式，是监测队列的消息数量。消费者的推荐编写方式是，在正确的处理完消息后再进行ack，那么在处理完成之前消息处于unacked状态，如果消费者忽然挂掉，消息会重新进入队列处于ready状态，所以，可以通过监测当前队列下ready和unacked消息的数量来判断，如果ready超过阈值，而unacked仍为0,说明此时消费者已经不存在了，需要重新创建一个消费者。
致于阈值的创建范围，则可以根据实际的需要情况来设置。
需要注意的是，如果采用AMQP的方式来获取队列中的消息数量，则只能获取当前队列中ready消息的数量，没法得知unacked状态的消息数量。这样，对于消息堆积的情况，就会没法区分究竟是由于消费者崩溃造还是由于消费者未能正确的ack消息所造成的。
如果Rest API queue检查的方式，则没有这方面的问题。

消息队列阈值基准经验法则

通过日志文件，我们可以获得合理而又准确的信息，再对信息进行分析，则可以确定队列的平均处理速度。
也可以通过图形监控系统来监控队列，来获取近乎精确的平均未消费和未确认消息总数。
一般认为：

  • 高于正常情况20%-90%，属于正常波动，可设为warning
  • 高于正常情况100%，则肯定已经出现了差错，应设为critical

重连

重新建立连接的方式，一般有两种，一种是程序直接退出，通过监控框架或程序来重启，再进行连接
一种是在程序中重试，尝试重新连接，此时原消费者不可用，应重新注册一个新连接下的消费者
