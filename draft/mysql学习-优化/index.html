数据库优化

数据库优化必要性，主要方式

explain分析

使用explain非常简单，只需要在查询语句前加上explain即可，但是需要注意的是，如果语句包含开销较大的子查询或使用临时表算法的视图，实际上会给服务器带来大量工作。 在MySQL5.6之后，允许解释非SELECT语句，如果使用的是5.6之前的版本，需要将语句改写成接近的SELECT语句后再进行分析。

explain列

  • id列 标识SELECT查询的序列号

  • select_type列 这一列显示了对应行是简单还是复杂的操作

  • table列 这一列显示了访问的对应表

  • type列 这一列很重要，显示了访问类型——简单讲就是MySQL决定如何查找表中的行。

    • ALL 全表扫描，性能最差
    • index 跟全表扫描类似，只是扫描表时按索引次序进行而不是行。主要优点就是避免了排序，主要缺点就是要承担按索引次序读取整个表的开销。这通常意味着若是按随机次序访问行，开销将会非常大。如果在Extra列中看到“Using index”，说明MySQL正在使用覆盖索引，它只扫描索引的数据，而不是按索引次序的每一行。它比按索引次序全表扫描的开销要少很多。
    • range 范围扫描，一个有限制的索引扫描，开始于索引里的某一点，返回匹配这个值域的行，这比全索引扫描好一些。

  • possible_keys列

  • key列

  • key_len列

  • ref列

  • rows列

  • filtered列

  • extra列
